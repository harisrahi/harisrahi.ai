A Step-by-Step Playbook for Mapping a Legacy Application with Codex and Claude Code

What this playbook is for

Use this playbook when an application is spread across several code repositories and nobody has a complete, trusted picture of how it all fits together.

You will use Codex and Claude Code to:

  1. Study the same repositories independently.
  2. Check each product's report against the source.
  3. Compare the checked reports to find meaningful differences.
  4. Resolve those differences into one final system map.
  5. Have developers review and accept the final map.

This work comes before modernization planning. The goal is to understand the current system as well as the available evidence allows.

The most important rule

An honest gap is better than a confident guess.

The reports must clearly separate:

  • What the source directly confirms.
  • What the source supports but does not prove.
  • What cannot be determined from the available repositories.
  • Where credible evidence conflicts.

Source code may show that an API connection has been built. It does not prove that the connection is deployed, enabled, or currently used in production.

Step 1: Prepare the workspace

What you are doing

Open the common parent folder containing all repositories in Codex and Claude Code. Give each product a different output folder.

That is the only access setup this playbook requires. The discovery prompt contains the rules about what the agents may read, where they may write, and what they must not change.

Why this matters

Both products need to examine the same code. Separate output folders prevent one product from reading or overwriting the other product's work during discovery.

Suggested folder structure

legacy-system/
  repository-one/
  repository-two/
  repository-three/

  architecture-analysis/
    codex-discovery/
    claude-discovery/
    claude-verifies-codex/
    codex-verifies-claude/
    comparison/
    final-system-map/

Keep the analysis folders outside the source repositories.

Choose the models

  • In Codex, use GPT-5.6 Sol at High or higher.
  • In Claude Code, use Claude Fable 5 at High or higher.

Keep the repositories stable

Start both discovery runs close together. If practical, ask developers not to pull changes, switch branches, or edit the repositories until both discovery runs finish.

The prompt tells each agent to record the current branch, the exact Git commit ID, and the Git status at the beginning and end of its work.

A Git commit ID identifies the exact saved version of a repository. Starting both products at the same time helps keep their views aligned. Recording the commit ID catches changes that happen without anyone noticing.

Git status records uncommitted changes. Comparing the starting and ending status gives developers a practical way to see whether files or Git state changed during the run. This check cannot prove that no command was ever attempted, but it can show whether the repository ended differently.

Check before moving on

  • Both products can see every repository.
  • Both products have their own output folder.
  • Both models are set to High or higher.
  • Developers know not to change the repositories during discovery when that is practical.

Step 2: Run independent discovery

Run the same prompt once in Codex and once in Claude Code. Do not give either product the other product's report.

What this prompt does

The discovery prompt asks each product to examine all the repositories and build an evidence-backed picture of the current system. It identifies what each repository contains, how the repositories connect, how data moves, and what the available source cannot establish.

Why you run it

Developers may understand individual parts of a legacy system without understanding the whole system. Two independent discovery runs reduce the chance that one model's assumptions or omissions become the accepted picture.

How it works

The agent inspects source code, configuration, tests, deployment files, infrastructure files, and relevant Git history. It connects important conclusions to specific evidence. It records uncertainty instead of filling gaps with plausible guesses.

The prompt separates coverage from gaps:

  • Coverage answers, "What did the agent actually examine?"
  • Gaps answer, "What did the agent examine but still could not determine?"

This prevents an unexamined area from being mistaken for proof that something is absent.

What you get

Each discovery run creates five required outputs:

  1. STATUS.md, so interrupted work can continue.
  2. SYSTEM_REPORT.md, the main explanation of the application.
  3. repos/[repository-name].md, supporting notes for each repository.
  4. COVERAGE.md, showing what was and was not examined.
  5. GAPS.md, showing unresolved questions and conflicting evidence.

Later agents can compare and verify the Markdown reports directly.

What you do next

Wait for both discovery runs to finish. Check that both products examined the same repositories and exact saved versions. Then start the separate verification sessions in Step 3.

How to run the prompt

  1. Start a new Codex session from the common parent folder.
  2. Select GPT-5.6 Sol at High or higher.
  3. Paste the prompt below and enter the Codex output folder.
  4. Start a new Claude Code session from the same parent folder.
  5. Select Claude Fable 5 at High or higher.
  6. Paste the same prompt and enter the Claude output folder.
  7. Do not share either discovery report with the other product yet.

Discovery prompt

# Goal

Build an accurate, evidence-backed map of the application represented by every in-scope Git repository under WORKSPACE_ROOT.

This is discovery only. Do not fix, refactor, modernize, or recommend changes.

WORKSPACE_ROOT: [ABSOLUTE PATH TO THE COMMON PARENT FOLDER]
OUTPUT_DIRECTORY: [ABSOLUTE PATH TO THIS PRODUCT'S DISCOVERY FOLDER]

# Accuracy rules

An honest gap is better than a confident claim that the available evidence cannot support.

For every important claim:

- Cite the repository, exact Git commit ID, file, and line range. Use the closest stable code symbol or configuration key when line numbers are unavailable.
- Label the claim Confirmed by source, Supported but not proven, Unknown, or Conflicting evidence.
- Use Confirmed by source only when the available source directly establishes the claim.
- Use Supported but not proven when evidence points to the claim but does not establish it fully.
- Use Unknown when the available evidence cannot answer the question.
- Use Conflicting evidence when credible sources disagree.
- Search for better evidence before accepting an inference. If the answer remains uncertain, record the gap.
- Say "not found in the areas examined," rather than "does not exist."
- Treat names, READMEs, comments, and diagrams as evidence of intent. Check them against implementation and configuration where possible.
- Do not quietly choose one side when evidence conflicts.

Report what the source implements. Do not claim that something is deployed, enabled, currently used, or receiving production traffic without suitable runtime evidence.

# Working boundaries

- Treat every source repository as read-only.
- Write only inside OUTPUT_DIRECTORY.
- Do not change source files, configuration, lockfiles, Git state, branches, commits, staging, or stashes.
- Use safe, read-only inspection commands.
- Do not connect to production systems, call live application APIs, or change any external system.
- You may use official technical documentation when it helps you understand a framework, library, or file format. Do not use internet sources as proof of how this application currently behaves.
- Do not build, run tests, install packages, or start services unless the user separately approves a clearly local investigation.
- Never display or copy secret values. Record configuration variable names and their apparent purposes only.
- Ask the user only about a blocker that cannot be resolved from the available repositories or safe inspection. Record non-blocking uncertainty and continue.

# How to work

Use a staged investigation. Keep STATUS.md current so the work can continue after interruption or context loss.

When subagents are available and useful, give them bounded, read-only investigations. Keep cross-repository synthesis, conflict review, and final quality control with the coordinating agent.

Do not report private reasoning. Report conclusions, evidence, conflicting evidence, searches performed, and unresolved questions.

# What to investigate

Establish:

1. Every repository found under WORKSPACE_ROOT, plus its path, branch, exact Git commit ID, starting Git status, main languages, and major directories.
2. The supported purpose of each repository, its entry points, deployable parts, main modules, interfaces, dependencies, configuration, and tests.
3. Connections between repositories through APIs, events, queues, jobs, shared databases, shared files, libraries, generated clients, authentication, builds, deployments, infrastructure, and matching configuration.
4. Which components read or write each database, schema, table, queue, cache, file store, or external store that can be identified.
5. Important end-to-end flows that can be traced through the available source.
6. Build, deployment, infrastructure, environment, and operating setup shown by the repositories.
7. Visible timeout, retry, fallback, and failure behavior for important connections.
8. Missing repositories, incomplete flows, conflicting contracts, stale documentation, unexamined areas, and questions requiring runtime evidence or human knowledge.

For an important connection between repositories, seek evidence at both ends when both ends are available. Record the source, target, direction, method, data or contract involved, when the connection occurs, evidence, confidence, and visible failure handling.

Do not call code unused or unreachable merely because you did not find a caller. State where you searched and what you did not find.

# Required outputs

Create only these required outputs inside OUTPUT_DIRECTORY:

1. STATUS.md
   Record completed work, remaining work, blockers, the last completed step, and the exact next step.

2. SYSTEM_REPORT.md
   Provide a plain-English system overview. Cover repository roles, system boundaries, important components, connections, data and state, end-to-end flows, authentication, build and deployment setup, infrastructure, operating behavior visible in source, and evidence-backed Mermaid diagrams. Clearly label confirmed findings, supported but unproven findings, unknowns, and conflicts.

3. repos/[repository-name].md
   Create one supporting report per repository. Cover purpose, entry points, deployable parts, modules, interfaces, dependencies, data, configuration, tests, build and deployment evidence, important history, and open questions.

4. COVERAGE.md
   Record every repository found, its path, branch, exact Git commit ID, starting and ending Git status, areas examined, areas skipped, exclusions, missing referenced repositories, and the search used for important absence claims.

5. GAPS.md
   Record questions investigated but not answered, conflicting evidence, missing repositories or systems, claims requiring runtime evidence, and claims requiring human knowledge. For each gap, state what was checked and what evidence would answer it.

Do not create additional deliverables unless one of these files would otherwise become unusable. If another file is necessary, explain why in STATUS.md.

# Completion check

Before finishing:

- Every repository found is analyzed or clearly marked blocked.
- Repository branches and exact Git commit IDs are recorded.
- Starting and ending Git status are compared.
- Important claims cite evidence and use the required labels.
- Cross-repository connections, data movement, build and deployment relationships, and end-to-end flows were actively investigated.
- Coverage and gaps are kept separate.
- The reports state what the available source could not establish.
- All five required outputs agree with one another.

Do not claim complete understanding when important repositories, configuration, runtime systems, or evidence are missing.

# Final response

In plain language, state what you established, where the map is weakest, which gaps matter most, and which report the reader should open first.

While the agents work

If an agent stops after planning or is interrupted, tell it:

Continue the discovery. Read STATUS.md and carry out the next recorded step. First confirm that the repository versions have not changed. Stop only if you need information or access that I must provide.

If you answer a factual scope question for one product, give the same answer to the other product when it applies.

Check before moving on

For both discovery folders, confirm:

  • All five required outputs exist.
  • COVERAGE.md lists the same repositories, branches, and exact Git commit IDs.
  • Every repository has supporting notes or is marked blocked.
  • GAPS.md separates missing evidence from unexamined areas.
  • The ending Git status matches the expected starting state.

If the products examined different versions, correct that before verification.

Step 3: Have each product check the other report

Claude Code will check the Codex report. Codex will check the Claude Code report. Use a new session for each verification.

What this prompt does

The verification prompt treats the discovery report as a set of claims that must be checked against the source. It checks whether the cited evidence supports the wording, whether confidence is too high, whether important findings were missed, and whether the stated coverage is honest.

Why you run it

A detailed report can still be wrong. Verification tests each report before the two reports are compared. This prevents the comparison stage from spending time comparing unsupported claims.

Using the other product as verifier also gives the report a fresh reader that did not create it.

A new session matters because it does not carry the discovery session's earlier assumptions and working conversation.

How it works

The verifier receives one discovery folder and the same repositories. It does not receive its own earlier discovery report. It checks material claims directly against the source and records corrections without rewriting the original report.

A material claim is one that would change the reader's understanding of repository purpose, system boundaries, important connections, data ownership, end-to-end flows, authentication, build and deployment, infrastructure, or major gaps.

What you get

Each verification produces one file: VERIFICATION_REPORT.md.

That file contains the snapshot check, an overall reliability judgment, a table of checked claims, missed findings, coverage problems, remaining unknowns, and a starting-to-ending Git status check.

What you do next

If either verifier finds major problems, have a new discovery session repair that report before continuing. Once both reports are suitable for comparison, move to Step 4.

How to run the prompt

In Claude Code:

  1. Start a new session using Claude Fable 5 at High or higher.
  2. Provide the Codex discovery folder and the repository workspace.
  3. Do not provide Claude's discovery folder.
  4. Use the claude-verifies-codex output folder.

In Codex:

  1. Start a new session using GPT-5.6 Sol at High or higher.
  2. Provide the Claude discovery folder and the repository workspace.
  3. Do not provide the Codex discovery folder.
  4. Use the codex-verifies-claude output folder.

Verification prompt

Use this prompt twice. Change the report and output paths.

# Goal

Check one discovery report against the exact repository versions it claims to have examined.

Treat every material claim as untrusted until the source supports it. Do not compare this report with another discovery report. Do not rewrite the report.

REPORT_DIRECTORY: [ABSOLUTE PATH TO THE DISCOVERY REPORT BEING CHECKED]
WORKSPACE_ROOT: [ABSOLUTE PATH TO THE COMMON PARENT FOLDER]
OUTPUT_DIRECTORY: [ABSOLUTE PATH TO THIS VERIFICATION FOLDER]

# Verdicts

Use one of these verdicts for each material claim:

- Supported
- Too broad
- Unsupported
- Contradicted
- Cannot verify from available source
- Citation problem
- Coverage overstated

# Rules

- Check claims against repository evidence, not architectural plausibility.
- Do not treat agreement between models as evidence.
- Do not claim live or production behavior without suitable runtime evidence.
- Say "not found in the areas checked," rather than "does not exist."
- Treat source repositories as read-only and write only inside OUTPUT_DIRECTORY.
- Do not connect to production systems, call live application APIs, or change external systems.
- Official technical documentation may explain a framework or dependency. It cannot prove how this application currently behaves.
- Do not expose secret values.
- Do not make modernization recommendations.

# First check

Compare the discovery report's repository list, branches, exact Git commit IDs, and recorded Git status with the available workspace.

If the repository versions do not match, record the mismatch and stop. The report cannot be fairly verified against different code.

# What to check

Read all five required discovery outputs.

For every material claim:

1. Open the cited evidence.
2. Decide whether it supports the exact wording and confidence.
3. Search for nearby or conflicting implementation, configuration, tests, contracts, deployment files, infrastructure files, and relevant history.
4. Check whether a README, comment, diagram, or name was treated as proof.
5. Check whether source capability was presented as deployed or current production behavior.
6. Check important repository connections from both ends when both ends are available.
7. Check whether an absence statement is supported by the documented coverage and search.
8. Record better evidence, conflicting evidence, narrower wording, or missing evidence.

Within the areas checked, look for important components, connections, data stores, flows, or conflicts the discovery report missed. Record these separately. Do not silently add them to the original report.

# Required output

Create only OUTPUT_DIRECTORY/VERIFICATION_REPORT.md with these sections:

1. Report and repository versions checked.
2. Overall reliability of the report.
3. Material claim audit table containing the report claim, report location, verdict, evidence checked, correction or narrower wording, and evidence still needed.
4. Important findings missed by the report.
5. Problems with citations, coverage, gaps, or confidence.
6. Claims the available source cannot settle.
7. Starting and ending Git status comparison.
8. Final judgment:
   - Suitable for comparison as written.
   - Suitable after applying the listed corrections.
   - Needs major discovery work before comparison.

Do not create other deliverables.

# Completion check

- Every material claim was checked or clearly marked as not checked with a reason.
- Corrections cite source evidence.
- Missed findings are separate from corrections.
- Claims requiring runtime or human evidence remain unresolved.
- Starting and ending Git status were compared.

Do not call the report verified merely because most citations are valid.

Check before moving on

  • Both VERIFICATION_REPORT.md files exist.
  • Both verifiers checked the same repository versions used during discovery.
  • Corrections include supporting evidence.
  • Missed findings are clearly listed.
  • Claims requiring runtime evidence remain unresolved.
  • Both reports are suitable for comparison, either as written or with the listed corrections.

Step 4: Compare the checked reports

Use a new session in either Codex or Claude Code. This session receives both discovery reports and both verification reports.

What this prompt does

The comparison prompt identifies where the checked reports agree, where one contains useful findings the other missed, where confidence differs, and where claims still conflict.

Why you run it

Verification asks whether each report is supported. Comparison asks how the two checked reports differ. Keeping these jobs separate stops model agreement from being mistaken for proof.

A new session keeps the comparison focused on the four completed reports instead of carrying assumptions from an earlier stage.

How it works

The comparison agent applies the verifier's corrections before comparing the reports. It compares meaning rather than writing style or report length. It does not inspect the repositories or decide which remaining claim is correct.

What you get

The comparison creates one file: COMPARISON_REPORT.md.

It explains the overall level of agreement and lists the exact conflicts, missing findings, coverage differences, and shared unknowns that the final session must address.

What you do next

Review the comparison report. If it shows that the products examined different repository versions or that verification was incomplete, fix that first. Otherwise, give the comparison report and all earlier reports to the final reconciliation session in Step 5.

How to run the prompt

  1. Start a new session in Codex or Claude Code.
  2. Use GPT-5.6 Sol or Claude Fable 5 at High or higher.
  3. Provide both discovery folders and both verification folders.
  4. Do not ask the agent to inspect the repositories during this step.
  5. Use the comparison output folder.

Comparison prompt

# Goal

Compare two independently created and independently verified discovery reports. Identify what the final reconciliation session must resolve.

Do not decide which remaining claim is correct. Do not inspect the repositories. Do not merge or rewrite the reports.

CODEX_REPORT_DIRECTORY: [ABSOLUTE PATH]
CLAUDE_REPORT_DIRECTORY: [ABSOLUTE PATH]
CLAUDE_VERIFICATION_DIRECTORY: [ABSOLUTE PATH]
CODEX_VERIFICATION_DIRECTORY: [ABSOLUTE PATH]
OUTPUT_DIRECTORY: [ABSOLUTE PATH TO THE COMPARISON FOLDER]

# Rules

- Apply the verification corrections before comparing claims.
- Compare meaning, not writing style, formatting, length, or level of detail.
- Do not treat agreement as proof.
- Do not treat omission as contradiction.
- Do not hide unresolved conflicts.
- Do not make modernization recommendations.

# Difference types

Use these plain-language types:

- Checked agreement: both corrected reports make the same supported claim.
- Compatible detail: one report adds supported detail without disagreeing.
- Missing from one report: one report contains a supported finding the other omitted.
- Different confidence: both describe the same thing with different certainty.
- Direct conflict: the corrected claims cannot both be right.
- Different coverage: the products examined different areas.
- Shared unknown: neither report could answer the question.
- Wording difference: the wording differs but the meaning is the same.

# What to compare

Compare:

- Repository lists and exact Git commit IDs.
- Repository purposes and deployable parts.
- System boundaries and responsibilities.
- Connections between repositories.
- APIs, events, queues, jobs, shared libraries, and generated clients.
- Data readers, writers, ownership, and shared state.
- Authentication and authorization boundaries.
- Build, deployment, infrastructure, and configuration relationships.
- End-to-end flows.
- Timeout, retry, fallback, and failure behavior.
- Coverage, gaps, conflicts, and findings added by the verifiers.

# Required output

Create only OUTPUT_DIRECTORY/COMPARISON_REPORT.md with these sections:

1. Plain-English summary of how closely the checked reports align.
2. Material difference table containing the Codex position, Claude position, verification correction, difference type, importance, why it matters, and evidence needed to settle it.
3. Important checked findings present in only one report.
4. Coverage examined by both products, one product, or neither.
5. Shared unknowns and the evidence needed to answer them.
6. Exact reconciliation checklist for the final session.
7. Overall reconciliation size: small, moderate, or substantial, with a short reason.

Do not create other deliverables.

# Final response

State how closely the checked reports align, the most important remaining differences, and whether final reconciliation appears small, moderate, or substantial.

Check before moving on

  • The comparison applies both verification reports.
  • Missing information is not called a contradiction.
  • Remaining conflicts are stated clearly.
  • Coverage differences and shared unknowns remain visible.
  • The reconciliation checklist points to exact claims or report sections.

Step 5: Create the final system map

Use a new session in Codex or Claude Code. Give it the repositories, both discovery reports, both verification reports, and the comparison report.

What this prompt does

The reconciliation prompt creates one final description of the system. It resolves the comparison checklist against the source, applies the verification corrections, and keeps anything that still cannot be settled clearly marked.

Why you run it

The two discovery reports will contain overlapping findings, different details, and possibly direct conflicts. The final map needs one supported version of each important claim. It must rely on evidence rather than model agreement or confidence.

A new session gives the final agent a clean starting point. It has no earlier report of its own to defend.

How it works

The agent reads every earlier report, then inspects the source for the items that remain different, uncertain, or important. It uses the narrowest wording the evidence supports. It keeps a claim unknown or conflicting when the evidence cannot settle it.

What you get

The final session creates three required files:

  1. FINAL_SYSTEM_MAP.md, the readable description of the current system.
  2. COVERAGE.md, the combined record of what was examined.
  3. GAPS.md, the remaining unknowns, conflicts, and evidence needed.

The final map includes repository roles, connections, flows, data, authentication, build and deployment, infrastructure, diagrams, and the important changes made through verification and reconciliation.

What you do next

Give the three final files to the developers and system owners for the human acceptance review in Step 6.

How to run the prompt

  1. Start a new session in Codex or Claude Code.
  2. Use GPT-5.6 Sol or Claude Fable 5 at High or higher.
  3. Provide the exact repository workspace.
  4. Provide both discovery folders, both verification folders, and the comparison folder.
  5. Add any available runtime evidence. Label where it came from and when it was collected.
  6. Use the final-system-map output folder.

Reconciliation prompt

# Goal

Create one final, evidence-backed map of the current system from two discovery reports, two verification reports, their comparison, and the available repositories.

Do not select a claim because both models agree, because one report is longer, or because one explanation sounds cleaner. Use source evidence and any supplied runtime evidence.

WORKSPACE_ROOT: [ABSOLUTE PATH TO THE COMMON PARENT FOLDER]
CODEX_REPORT_DIRECTORY: [ABSOLUTE PATH]
CLAUDE_REPORT_DIRECTORY: [ABSOLUTE PATH]
CLAUDE_VERIFICATION_DIRECTORY: [ABSOLUTE PATH]
CODEX_VERIFICATION_DIRECTORY: [ABSOLUTE PATH]
COMPARISON_DIRECTORY: [ABSOLUTE PATH]
ADDITIONAL_RUNTIME_EVIDENCE: [PATHS OR NONE]
OUTPUT_DIRECTORY: [ABSOLUTE PATH TO THE FINAL SYSTEM MAP FOLDER]

# Accuracy rules

- Every important final claim needs evidence.
- Label claims Confirmed by source, Supported but not proven, Unknown, or Conflicting evidence.
- Do not claim that something is deployed, enabled, currently used, or receiving production traffic without suitable runtime evidence.
- Agreement between models is useful context, not proof.
- Omission is not contradiction.
- Do not infer absence from an area nobody examined.
- Keep conflicts visible when the available evidence cannot settle them.
- Do not make modernization recommendations.

# First checks

1. Confirm that discovery and verification used the same repository versions.
2. Confirm that both verification reports and the comparison report are complete.
3. Record any additional runtime evidence, where it came from, and when it was collected.
4. If repository versions do not match, stop and explain the problem.

# How to reconcile

For every important claim and every item in the comparison report's reconciliation checklist:

1. Find the claim in both discovery reports when present.
2. Apply the relevant verification findings.
3. Inspect the cited source and any conflicting source.
4. Inspect supplied runtime evidence when it is relevant and trustworthy.
5. Use the narrowest wording the evidence supports.
6. Keep the claim Unknown or Conflicting evidence when the available evidence cannot decide.
7. Record why a material claim was removed, narrowed, added, or left unresolved.

Also check high-consequence claims on which the reports agree. Agreement alone does not prove them.

# Required outputs

Create only these files inside OUTPUT_DIRECTORY:

1. FINAL_SYSTEM_MAP.md
   Provide a plain-English system overview. Cover the role of every repository, system boundaries, deployable parts, important connections, data and state, end-to-end flows, authentication, build and deployment, infrastructure, operating behavior supported by evidence, and evidence-backed Mermaid diagrams. Include a short section listing material claims removed, narrowed, added, or left unresolved during verification and reconciliation.

2. COVERAGE.md
   Record the final repository list, paths, branches, exact Git commit IDs, areas examined by discovery and verification, areas examined by only one product, areas examined by neither, exclusions, starting and ending Git status, and limits of the investigation.

3. GAPS.md
   Record every important unanswered question, unresolved conflict, missing repository or external system, claim requiring runtime evidence, and claim requiring human knowledge. For each gap, state what was checked and what evidence would answer it.

Do not create JSON or any other required deliverable.

# Final checks

- Every item in the reconciliation checklist is resolved or kept visibly unknown.
- Verification corrections are applied.
- Important claims cite evidence.
- High-consequence agreed claims were checked.
- Live behavior is not claimed from source evidence alone.
- Coverage and gaps remain separate.
- Starting and ending Git status are compared.
- The three final files agree with one another.
- No modernization recommendations appear.

# Final response

In plain language, state what the evidence establishes, what changed through verification and reconciliation, the most important remaining gaps, and which final file the reader should open first.

Check before moving on

  • All three final files exist.
  • Every comparison checklist item is resolved or remains clearly unknown.
  • Verification corrections appear in the final map.
  • Important claims link to evidence.
  • Claims about live use rely on runtime evidence, not source code alone.
  • Coverage and gaps remain separate.
  • The ending Git status matches the expected starting state.

Step 6: Human acceptance review

What you are doing

Developers and system owners are deciding whether the final system map is trustworthy enough to use as the shared description of the current application.

They do not need to reread every line of code. They should check the major system relationships, test disputed claims, and confirm that important limits remain visible.

Why this matters

The agents cannot create missing runtime evidence or replace the knowledge of people who operate the system. A human must decide whether the map is complete and reliable enough for its intended use.

What to read

Read in this order:

  1. FINAL_SYSTEM_MAP.md
  2. GAPS.md
  3. COVERAGE.md

Use the original discovery and verification reports only when you need to trace how a conclusion was reached.

Acceptance checklist

  • Are all expected repositories included?
  • Did both discovery runs and both verification runs use the intended repository versions?
  • Do the main repository roles and system boundaries match developer understanding?
  • Are important connections backed by source evidence?
  • Are claims about live use backed by runtime evidence?
  • Are missing repositories and external systems visible?
  • Are unexamined areas, unanswered questions, and conflicting evidence easy to find?
  • Do the diagrams match the written explanation?
  • Were the verification corrections applied?
  • Were modernization recommendations kept out?
  • Does the final Git status match the expected starting state?

If someone disagrees with the map

Record:

  • The exact claim they disagree with.
  • Their proposed correction.
  • The evidence supporting their correction.
  • Whether that evidence can be inspected.

Use the explanation to locate and check evidence. If supporting evidence is unavailable, label the correction as human-provided information. Do not silently replace an evidence-backed claim with an unsupported recollection.

Acceptance outcomes

Choose one:

  • Accepted: The map is suitable as the evidence-backed description of the available system.
  • Accepted with open questions: The map is useful, but the listed gaps must remain visible.
  • More evidence required: Important repositories, runtime evidence, or human knowledge are still missing.
  • Repeat part of the process: A discovery, verification, comparison, or reconciliation step was incomplete or used the wrong repository version.
